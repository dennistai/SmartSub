# 中／英／泰 語音 → 統一繁體中文（含原始語言標註）處理規則

> 適用情境：一段錄音／字幕中夾雜**繁體中文、簡體中文、泰文、英文**，最終要輸出**統一繁體中文**，並對「由英文或泰文轉譯而來」的句子標註原始語言。
> 本文件為實作無關（language-agnostic）的規格，可直接用於其他專案。以 SmartSub 的實測結果為依據。

---

## 0. 名詞與語言碼

| 名稱     | 碼                         | 說明                     |
| -------- | -------------------------- | ------------------------ |
| 繁體中文 | `zh-Hant`                  | 最終輸出字形（樞紐目標） |
| 簡體中文 | `zh-Hans` / `zh` / `zh-CN` | 需轉為繁體               |
| 泰文     | `th`                       | Unicode U+0E00–U+0E7F    |
| 英文     | `en`                       | 拉丁字母                 |

---

## 1. 整體流程（兩大階段）

```
錄音／音檔
   │  ── 階段一：多語言轉錄（ASR）──────────────
   ▼
逐句字幕（每句帶「偵測到的原始語言」標記：th / zh / en）
   │  ── 階段二：逐句轉繁中 + 標註 ─────────────
   ▼
統一繁體中文字幕（英/泰句下方另起一行標註原始語言）
```

- **階段一**負責「把混合語音正確轉成各語言文字」——這是最難、品質上限所在。
- **階段二**負責「把各語言統一成繁體中文並標註」——邏輯明確、確定性高。

> 若輸入已是文字（非語音），可略過階段一，直接進階段二。

---

## 2. 階段一：多語言轉錄（ASR）

### 2.1 核心限制（務必理解）

單次 Whisper（含 `large-v3-turbo`）對「整檔」只會**鎖定一種語言**，其餘語言會被轉錯或產生幻覺：

| 設定                      | 結果（實測 39 分鐘中泰英會議）                            |
| ------------------------- | --------------------------------------------------------- |
| `language=auto`           | 依開頭語言鎖定全檔；medium 甚至對其餘 37 分鐘吐幻覺佔位符 |
| `language=zh`（強制中文） | 中文段良好，但**泰文段被硬塞成中文亂碼**（泰文命中 = 0）  |

**結論：不能用「整檔單次轉錄」處理多語音檔。**

### 2.2 可行方案：分段 + 逐段語言偵測

把音檔切成短片段，**每段各自 `auto` 轉錄**。短片段通常單一語言，`auto` 偵測準確，且引擎會回報該段偵測到的語言。

```
for 每個 chunk（切片）:
    result = ASR(chunk, language='auto', vad=on, token_timestamps=on)
    lang   = result.language          # 引擎回報的偵測語言（可靠）
    conf   = mean(token.p)            # 該段平均 token 機率（信心，見 2.4 警告）
    cues  += result.segments（時間軸加回 chunk 絕對起點，並標記 lang）
```

實測（large-v3-turbo, RTX 4060）：

- 逐段 `auto` **成功救回單次轉錄完全漏掉的泰文**（例：泰國廠員工用泰文報告產線人力、ABC 三組輪替）。
- 速度：~30s/chunk、全程約 5–7 分鐘（≈ 單次轉錄速度，效能非瓶頸）。

### 2.3 切段（chunking）規則

- **優先用神經 VAD（如 silero）在語音／靜音邊界切**，避免切斷詞。
  - 注意：以 dB 閾值的靜音偵測（ffmpeg `silencedetect`）對**噪音底大的會議錄音會失效**（實測完全找不到靜音）；改用神經 VAD。
- 目標 chunk 長度 **~25–30 秒**（兼顧語言偵測穩定與 encoder 效率）。
- chunk 內開引擎自帶 VAD，跳過內部靜音。

### 2.4 信心（confidence）的重要警告

- 引擎的 token 平均機率**不能可靠區分「正確」與「自信地錯」**。實測 mis-detect 成英文的亂碼段，信心仍高達 0.83–1.00。
- 因此**不要只用「低信心」當作 mis-detect 的判準**。

### 2.5 語言誤判的修正（實測：文字啟發式有天花板）

mis-detect 的典型症狀：一段中文/泰文語音被 `auto` 判成英文，並轉出**英文亂碼**（真英文單字排成無意義句）。

**已實測、且證實不足的做法**（供避坑）：

1. **絕對信心門檻**（token 平均機率 < 0.55 才重轉）→ 無效。whisper 對誤判「自信地錯」，亂碼段信心常達 0.83–1.00。
2. **字元重複率門檻**（重複高 → 判幻覺）→ 部分有效但脆弱。一旦開啟抗重複參數，亂碼重複率被壓到 0.24–0.34（< 0.5 門檻），偵測失效；且「Put some kind T-suit need for some some go…」這類**非重複亂碼**根本抓不到。

**根因**：亂碼是「真英文單字 + 無意義語序」，來自會議前段的**多人交錯講話（crosstalk）**；單流 ASR 對此先天困難。文字層面（信心、重複、字典命中）都無法可靠區分「真英文」與「crosstalk 亂碼」。

**正解（已實測驗證有效）：專用「音訊語言辨識（audio LID）」逐段判語言**，取代「auto 轉錄後看回報語言」。

- **實測用 `facebook/mms-lid-256`（transformers, Wav2Vec2, CPU 可跑）**：
  - 把 whisper 誤判成英文亂碼的 crosstalk 段（@90s）正確判為 **zh（0.93）**；乾淨中文段 **zh（0.99）**；泰文段 **th（0.999）**。
  - 全檔 chunk 語言分佈：LID = **th 15 / zh 61 / en 2**（whisper auto 為 th 0、大量假 en）。
  - 用 LID 語言逐段轉錄後：**英文亂碼歸零**（剩餘 en 全是真英文），**泰文回收翻倍以上**。
- **流程**：先用 audio LID 決定每段語言（限定到 {th, zh, en} 取最高分，化解 Thai↔Burmese/Lao 混淆）→ 再用該語言以 whisper.cpp 轉錄（並可**依語言選最適模型**，見 §2.6）。
- 其他可選 LID：SpeechBrain VoxLingua107、`facebook/mms-lid-*` 其他尺寸、或暴露 Whisper `whisper_lang_auto_detect` 的**機率分佈**做門檻/投票。
- **殘留限制**：chunk 內若真有語碼交錯（同 30s 內泰語員工與中文 GM 交疊），chunk 級 LID 只能取主導語言；需更細的 VAD 對齊切段才能再改善。

輔助（可與 LID 併用）：鄰居語言平滑（孤立語言段參考 i±2 鄰居主導語言）、whisper 抗幻覺參數（`condition_on_previous_text=false`、`no_repeat_ngram_size=3`、`repetition_penalty=1.1`）降低源頭重複。

### 2.6 依語言路由模型（實測，推薦）

LID 決定每段語言後，不必全檔用同一個 ASR 模型——**逐段依語言選最適模型**能再進一步。SmartSub 窗口級實測（三語各取代表片段，比較 Breeze-ASR-25 vs large-v2 vs large-v3-turbo，詳見 `breeze-vs-whisper-comparison.md`）：

| 語言                       | 最適模型                                           | 說明                                                                                                                                                                                              |
| -------------------------- | -------------------------------------------------- | ------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------- |
| 中文（含台灣腔／中英夾雜） | **Breeze-ASR-25**（聯發科，Whisper-large-v2 微調） | 內容最完整、貼近台灣口語；`auto` 下也**不會像 turbo/large-v2 跳成英文**（code-switching 強項）                                                                                                    |
| 英文                       | `large-v3-turbo`                                   | 長句與 Breeze 同級；Breeze 在資訊量極低的極短片段偶爾補幻覺                                                                                                                                       |
| **泰文**                   | `large-v3-turbo` / `large-v2`                      | **絕不可用 Breeze**：微調造成對泰語**災難性遺忘**——強制泰語仍破碎、混入拉丁/漢字；`auto` 更會**誤判成中文並吐幻覺翻譯、泰文整段消失**。母模型 large-v2 在 `auto` 下能正確停在泰文，證明是微調退步 |

- **實作**：同一 whisper.cpp addon 可用不同 `model` 路徑逐段轉錄不同 ggml 模型，故零額外成本。僅當路由目標模型**已安裝**時才切換，否則回退基礎模型（優雅降級，只裝一個模型時行為不變）。
- **關鍵**：因管線已用 LID 以**強制語言**（非 `auto`）呼叫 whisper，天然避開 Breeze 的自動偵測翻車；泰文段再路由到 turbo，即可兼得「中文用 Breeze、泰文不退步」。
- **一句話**：`zh → Breeze、th/en → turbo`。對以台灣中文為主、夾雜泰英的會議，這是目前最佳組合。

### 2.7 已知限制

- **句內語碼轉換（code-switching，一句話中英泰混講）**：分段法只能取該段主要語言，仍會有交界誤判。可接受，但無法完美。（Breeze 對「中文句內夾英文詞」的穩定度明顯較佳，但整句仍以主導語言為準。）
- 這是多語 ASR 的先天難點，非本規則能完全解決；階段一的目標是「大幅改善」而非「完美」。

## 3. 階段二：逐句語言判定（腳本分類）

當每句的語言**已由階段一標記**時，直接用該標記（最可靠）。
若沒有語言標記（例：輸入是純文字字幕），改用**文字腳本判定**：

### 3.1 Unicode 區塊（三者互斥）

| 語言      | Unicode 範圍                                                          |
| --------- | --------------------------------------------------------------------- |
| 泰文 `th` | U+0E00–U+0E7F                                                         |
| 中文 `zh` | U+4E00–U+9FFF（基本）、U+3400–U+4DBF（擴充 A）、U+F900–U+FAFF（相容） |
| 英文 `en` | U+0041–U+005A、U+0061–U+007A（A–Z a–z）                               |

### 3.2 主要腳本演算法

```
function classifyScript(text):
    逐 code point 計數三桶：thai / cjk / latin
    忽略：數字、空白、各類標點（ASCII 與全形）   # 不帶語言資訊
    max = max(thai, cjk, latin)
    if max == 0: return 'other'                   # 純數字/符號/空白
    return 對應 max 的語言（並列時 泰 > 中 > 英）
```

- **中文簡繁不在此判斷**：中文一律走 OpenCC 轉繁（見 4.1），已是繁體則不動。
- **限制**：短的混排句（如 `OK 為了GM`，拉丁字母數 > 中文）會被誤判為英文。**故階段一的語言標記優先於腳本判定。**

## 4. 階段三：逐句轉繁中規則

| 偵測語言           | 處理                                       | 是否標註                 |
| ------------------ | ------------------------------------------ | ------------------------ |
| **中文 (zh)**      | OpenCC 簡→繁；若已是繁體則不變。**不翻譯** | 否                       |
| **英文 (en)**      | 翻譯 `en → zh-Hant`                        | 是：`（原始語言：英文）` |
| **泰文 (th)**      | **兩段式** `th → en`，再 `en → zh-Hant`    | 是：`（原始語言：泰文）` |
| **其他/數字/空白** | 原樣保留                                   | 否                       |

### 4.1 中文：簡→繁（確定性）

- 用 OpenCC（詞組級）：`Converter({ from: 'cn', to: 't' })`（簡→繁標準）。
- 僅在「偵測到相反字形」時實際改寫（轉換前後不同才算命中），避免無謂改動。
- 對整段 SRT 安全：序號／時間碼／`-->` 皆 ASCII，OpenCC 不動。

### 4.2 英文：`en → zh-Hant`

- 直接呼叫翻譯服務（建議 LLM），來源語言 `en`、目標 `zh-Hant`。
- 譯出的中文再過一次 OpenCC 繁化（兜底 LLM 偶爾吐簡體）。

### 4.3 泰文：強制兩段式 `th → en → zh-Hant`

1. **第一段** `th → en`：得到英文中介文字（僅供第二段與除錯，不寫入輸出檔）。
2. **第二段** `en → zh-Hant`：把中介英文翻成繁中。
3. 譯出再過 OpenCC 繁化。

- **為何兩段式**：泰→中直翻品質不穩；以英文為樞紐可顯著改善（業務需求指定）。
- 若用 LLM，可一次呼叫完成；若用傳統翻譯 API，需兩次呼叫，且需確認平台支援 `th→en`、`en→zh-Hant` 兩語向（不支援則改用 LLM 供應商）。
- **失敗處理**：第一段失敗（回傳失敗佔位）時，跳過第二段、保留佔位、不加標註。

## 5. 原始語言標註規則

- 僅**由英文或泰文轉譯而來**的句子才標註；中文句（含簡轉繁）**不標註**。
- 格式：譯文**下方另起一行**。
  - 英文來源 → `（原始語言：英文）`
  - 泰文來源 → `（原始語言：泰文）`

範例（SRT，onlyTranslate 模式）：

```
3
00:00:05,000 --> 00:00:07,000
你好，歡迎觀看
（原始語言：英文）
```

雙語模式時，註解掛在「譯文區塊」下：

- `原文＋譯文`：`原文 / 譯文 / 註解`
- `譯文＋原文`：`譯文 / 註解 / 原文`

---

## 6. 後處理順序（關鍵，易錯）

對每句：

```
1. 取得本文 body：
     - 中文/其他：body = 原文
     - 英文/泰文：body = 譯文（失敗時用佔位/原文兜底）
2. 繁化：body = OpenCC(body → 繁體)            # 只作用於「本文」
3. （可選）中文標點清理：body = 去中文標點(body)
4. 附加註解：final = body + '\n' + 註解         # 注解在「繁化與標點清理之後」才加
```

- **順序不可顛倒**：若先加註解再做「中文標點清理」，會把註解的全形括號 `（）` 與冒號 `：` 一併清掉。**註解必須最後加、且不再經過標點清理。**

## 7. 邊界情況

| 情況                                  | 處理                                                                        |
| ------------------------------------- | --------------------------------------------------------------------------- |
| 全中文檔（無泰/英）                   | 只做簡→繁；零翻譯、零標註                                                   |
| 純數字／符號／空白句                  | 原樣保留，不翻譯、不標註                                                    |
| 空字串                                | 直接輸出空                                                                  |
| 句內中英泰混排（單句 code-switching） | 取主要腳本/主要語言；少數交界句可能歸錯（已知限制）                         |
| 短句含英文 token（`OK`、人名、`GM`）  | 腳本判定會誤判為英文 → 優先用階段一語言標記；或設「需含完整英文句」才判英文 |
| 輸出格式 srt/vtt/ass                  | 註解為多行 cue 內一行，安全                                                 |
| 輸出格式 lrc/txt                      | 可能壓平多行 → 註解行需確認是否保留（已知風險）                             |

## 8. 建議參數／門檻

| 項目             | 建議值                                                                                                                                 |
| ---------------- | -------------------------------------------------------------------------------------------------------------------------------------- |
| ASR 模型         | 依語言路由：`zh → Breeze-ASR-25`、`th/en → large-v3-turbo`（見 §2.6）；單模型時用 `large-v3-turbo`（比 medium 明顯更準、且不幻覺鎖定） |
| chunk 長度       | 25–30 秒，VAD 邊界對齊                                                                                                                 |
| 語言平滑鄰居視窗 | ±2 個 chunk                                                                                                                            |
| 重轉判準         | 用「比較 conf」而非「絕對低 conf」（後者會漏掉高信心亂碼）                                                                             |
| OpenCC           | 簡→繁：`from:'cn', to:'t'`；繁→簡：`from:'t', to:'cn'`                                                                                 |
| 樞紐目標語言     | 一律 `zh-Hant`（不論表單原目標語言）                                                                                                   |

## 9. 附錄

### 9.1 LLM 翻譯提示詞（可用變數 `${sourceLanguage}`、`${targetLanguage}`、`${content}`）

- 系統提示：指定「資深字幕翻譯專家，將 `${sourceLanguage}` 字幕譯成 `${targetLanguage}`，口語、簡潔、逐條獨立、只輸出 JSON、不加解釋」。
- 中文目標務必在提示詞中明確為「繁體中文」（避免「中文」歧義導致簡繁混雜）。
- 批次輸入建議 JSON：`{ "id": "原文", ... }`，輸出同鍵，只換值；並校驗「回傳條數 = 請求條數」避免與時間軸錯位。

### 9.2 端到端輸入／輸出範例

輸入（混合來源）：

```
繁中：歡迎觀看本教學
簡中：欢迎观看本教程
英文：Hello, welcome to this tutorial.
泰文：สวัสดีครับ ยินดีต้อนรับ
```

輸出（統一繁體中文 + 標註）：

```
歡迎觀看本教學

歡迎觀看本教程

你好，歡迎觀看本教學
（原始語言：英文）

您好，歡迎
（原始語言：泰文）
```
